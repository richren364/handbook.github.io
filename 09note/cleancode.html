# 1、整洁代码

# 1.1 要有代码

代码、模型和需求

  • 代码不再是问题，应当关注模型和需求？代码会自动生成。
  • 永远抛不掉代码，因为代码代码呈现了需求的细节。
  • 期望语言抽象程度继续提升，领域特定语言的数量继续增加。
  • 代码永存。

# 1.2 糟糕的代码

  • 杀手应用：发布周期开始拉长，缺陷总是不能修复。
  • 为什么要写糟糕的代码？
    • 快点完成需求？
  • 烂程序能运行，回头再清理

    • Later equals never.(稍后等于永不！) -- LeBlanc 法则

# 1.3 混乱的代码

  • 混乱的增加，团队生产力持续下降。生产力下降，增加人手。新人只会制造更多的混乱。

# 1.4 思想流派

  • 任何门派都并非绝对正确。
  • 本书是对象导师(Object Mentor)整洁代码派的说明。
  • 很多建议都存在争议。

# 1.5 我们是作者

  • 作者都有读者，作者有责任与读者做好良好的沟通。
  • 花费读与写时间的比例超过10:1。
  • 想要干得快，想要早点做完，想要轻松写代码，先让代码易读。

# 1.6 童子军军规

美国童子军一条简单军规

让营地比你来时更干净。

  • 每次清理一点坏味道。
  • 清理并不一定要花多少功夫：一个变量名、拆分一个过长的函数、消除一点重复代码、清理一个嵌套if语句。

# 1.7 前传与原则

  • 2002.PPP.（Agile Software Development: Principles, Patterns, and Practices）（《敏捷软件开发：原则、模式与实践》）
    • 关注面向对象设计的原则，专业开发者采用的实践方法。
    • 主张在代码层面与本书回响。
  • 对设计原则的引用
    • 单一权则原则(Single REsponsibility Principle, SRP)
    • 开放封闭原则(Open Closed Principle, OCP)
    • 依赖倒置原则(Dependency Inversion Principle, DIP)
    • 。。。
